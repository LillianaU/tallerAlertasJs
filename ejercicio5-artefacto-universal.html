Ejercicio 5

El Artefacto Universal

⏱ 30 min
🎯 Objetivo

Comprender la adaptabilidad a diferentes dispositivos (iPhone/iPad) y las convenciones de plataforma iOS, enfocándose en cómo un mismo componente — el action sheet — se transforma en un popover según el contexto de pantalla.

Conceptos Clave
Dos Componentes, Una Intención
📋

Action Sheet (iPhone)

Hoja modal que emerge desde la parte inferior de la pantalla. Ideal para espacios reducidos. El usuario lo descarta deslizando o tocando fuera.

🔲

Popover (iPad)

Ventana flotante anclada al elemento que la disparó. Aprovecha el espacio amplio del iPad y permite interacción sin bloquear el contexto.

⚙️

popoverPresentationController

API de UIKit que gestiona automáticamente la transición entre estilos de presentación según el tamaño de pantalla (size classes).

📐

Size Classes

Sistema de iOS (compact / regular) que permite adaptar layouts sin if/else. iPhone → compact width; iPad → regular width.

Comparativa
Action Sheet vs Popover
Característica Action Sheet Popover
Dispositivo típico iPhone compact width iPad regular width
Posición Anclado al borde inferior Flotante, anclado al elemento origen
Bloqueo de UI Sí — fondo oscurecido No — contexto visible detrás
Descarte Tap fuera · swipe down · Cancelar Tap fuera del popover
Scroll interno Sí, si hay muchas opciones Sí, configurable por altura
API clave UIAlertController(.actionSheet) popoverPresentationController
Flecha indicadora No Sí — señala el anchor
Implementación
Cómo la librería maneja las diferencias
Swift · UIKit 
// ① Crear el action sheet universal
let alert = UIAlertController(
    title:   "Agregar Archivo",
    message: nil,
    preferredStyle: .actionSheet  // siempre .actionSheet
)

// ② Agregar acciones
alert.addAction(UIAlertAction(title: "📷 Tomar Foto",     style: .default) { _ in })
alert.addAction(UIAlertAction(title: "🖼️ Elegir Galería",  style: .default) { _ in })
alert.addAction(UIAlertAction(title: "📁 Explorar Archivos",style: .default) { _ in })
alert.addAction(UIAlertAction(title: "Cancelar",            style: .cancel) { _ in })

// ③ Configurar popover para iPad
// UIKit lanza NSException si no se configura en iPad
if let popover = alert.popoverPresentationController {
    popover.barButtonItem  = addBarButton   // ← anclar al botón
    // — ó alternativamente —
    popover.sourceView      = addButton       // vista origen
    popover.sourceRect      = addButton.bounds
    popover.permittedArrowDirections = [.down, .up]
}

// ④ Presentar — iOS decide automáticamente
// iPhone → slide-up action sheet
// iPad   → floating popover con flecha
present(alert, animated: true)
Flujo en tiempo de ejecución
¿Qué pasa internamente?
1. present() es llamado

UIKit recibe la petición de presentación y consulta el traitCollection del view controller actual.

2. Evaluación de Size Class

Si horizontalSizeClass == .compact (iPhone) → action sheet modal. Si es .regular (iPad) → popover.

3. Advertencia en iPad sin anchor

Si popoverPresentationController no tiene sourceView ni barButtonItem, UIKit lanza una excepción en tiempo de ejecución.

4. Renderizado diferenciado

iPhone recibe una hoja modal animada desde abajo. iPad recibe un popover flotante con flecha apuntando al anchor configurado.

Debate
¿De quién es la responsabilidad?
📦 La Librería

Abstrae la complejidad

Argumenta que una sola API (.actionSheet) debería manejar automáticamente ambos casos. El desarrollador no debería preocuparse por size classes. Pero: necesita el anchor en iPad, lo cual exige conocimiento del contexto.

📱 El Desarrollador

Conoce el contexto

Es quien sabe desde qué elemento se dispara la acción. Debe configurar el anchor del popover. Si no lo hace, la app crashea. La responsabilidad es compartida pero el punto de falla es del desarrollador.

🎨 El Diseñador UX

Define la experiencia

Debe especificar explícitamente el comportamiento en iPad en los design specs. Un wireframe solo para iPhone es incompleto. La responsabilidad empieza en el diseño, no en el código.

💡 Conclusión del Ejercicio

iOS no tiene "una solución universal" — tiene patrones adaptativos. El popoverPresentationController es el puente que UIKit ofrece. La responsabilidad de una buena UX es triangular: librería + desarrollador + diseño deben alinearse para que el Artefacto Universal funcione en cualquier pantalla.
